# cs340

I learned that maintainable, readable, and adaptable programs are built by separating responsibilities instead of putting everything into one large file. One of the biggest examples of this was using the CRUD Python module from Project One as the database layer for the dashboard in Project Two. Instead of writing MongoDB connection logic directly inside every dashboard callback, the dashboard imported the AnimalShelter class and used its read method to retrieve records from the animals collection. This made the dashboard easier to understand because the CRUD module handled the database communication, while the Dash code handled the user interface, filters, data table, chart, and map.

The advantage of working this way is that the code became more reusable and easier to update. If the database connection needed to change, I could update the CRUD module instead of rewriting the entire dashboard. If I wanted to create another dashboard in the future, I could reuse the same CRUD structure and adjust the queries for a different client, database, or collection. This module could also be used in future applications that need to create, read, update, or delete records from MongoDB, such as an inventory system, customer management system, or another data dashboard.

When approaching this problem as a computer scientist, I started by focusing on what Grazioso Salvare needed from the application. They needed a way to identify animals that could be good candidates for different types of search-and-rescue training. Then I broke the project into smaller parts: connecting to MongoDB, retrieving the animal data, creating rescue type filters, displaying the data in a table, adding a breed distribution chart, and showing animal locations on a map. I also had to make sure the Reset Dashboard button returned all widgets to the original unfiltered state, including the filter selection, table, chart, and map.

This project was different from previous assignments because it felt more like a complete client/server application instead of a single coding exercise. I had to think about the database, the backend CRUD module, the frontend dashboard, and the user experience all at the same time. The dashboard needed to work correctly, but it also needed to be clear and easy for the client to use. In the future, I would use the same strategy by first identifying the client’s needs, reviewing the available data, planning the database queries, and then building an interface that presents the data in a useful way.

Computer scientists solve problems by designing systems that collect, organize, process, and present information. This matters because many companies have large amounts of data, but that data only becomes valuable when people can use it to make better decisions. In this project, the dashboard could help a company like Grazioso Salvare work more efficiently by filtering animal records based on rescue training criteria. Instead of manually searching through thousands of records, the organization could use the dashboard to quickly narrow down possible rescue dog candidates.

My work on this project could help Grazioso Salvare save time, reduce user error, and make the animal selection process more consistent. The rescue filters, data table, breed chart, and geolocation map all support the client’s goal of identifying useful animal records more quickly. This project showed me how databases and software applications can support real-world work by turning raw data into something organized, visual, and actionable.
